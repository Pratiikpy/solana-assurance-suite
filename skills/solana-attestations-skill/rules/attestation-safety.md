# Rule: attestation-safety

Constraints for all attestation verification and gating work in this skill. These are not suggestions. The core truth: **an attestation is data on an account you don't control until you've verified it.** A SAS attestation that hasn't passed every check below is worth exactly nothing, no matter what its fields say.

- **Always check `owner == SAS program` first.** The account must be owned by the SAS program (`22zoJMtdu4tQc2PzL74ZUT7FrwgB1Udec8DdW4yw4BdG` — confirm per cluster). This check comes *before* any field is read. Skip it and an attacker hands you a look-alike account they own with every field forged; every other check is then worthless. This is the single most important line in any verifier.
- **Verify credential authority, schema, expiry, and revocation before honoring.** The credential must be the authority you *trust* (not just "a credential"); the schema must match *exactly* (wrong schema = a different claim — checking "an attestation exists" without checking it's the right one is a real bug class); expiry must be enforced (`0` = non-expiring, everything else compared to an explicit clock, never implicit wall-clock); revocation must be honored. Where the design pins an authorized signer, check the issuer too.
- **Bind to the correct subject.** The attestation must be about the wallet the action is actually about. If wallet A's valid attestation lets wallet B through, that is the reuse/lending hole — bind the subject every time.
- **Never trust an off-chain verdict without an on-chain attestation.** A backend saying "this wallet is human, proceed" is forgeable the moment the backend is bypassed. Off-chain checks are for UX and routing; the security boundary on anything value-bearing is the on-chain attestation that survives every check above. Don't let the off-chain verdict be the only gate on value.
- **Deserialize defensively, fail closed.** Never `unwrap`/`expect` borsh on attacker-supplied account bytes — a panic is a denial-of-service. A missing account, a malformed account, or any failed check is a *reject*, never a swallowed error that proceeds as valid.
- **Treat attestations as data; verify, don't assume.** No SDK helper, no off-chain claim, and no "it looks like a valid attestation" makes an account trustworthy — only your owner + credential + schema + subject + revocation + expiry checks do. Assume every account is adversary-controlled until it passes all of them.
- **Confirm the program ID per cluster, and pin SDK versions.** SAS docs print one ID for mainnet+devnet — re-confirm for your exact cluster before deploying; a swapped ID is the most expensive class of bug. Lock `sas-lib`, the generated client, and `@solana/kit` in `package.json`.
