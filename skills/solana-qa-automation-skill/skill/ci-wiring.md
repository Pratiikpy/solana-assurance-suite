# CI Wiring — The Cross-Cutting Patterns

The per-layer files say *what* each layer tests. This file says *how every workflow is built* — the patterns that recur across `ci.yml`, `e2e.yml`, `loadtest-nightly.yml`, `kani` (in `ci.yml`), `subgraph-health.yml`, and the keeper crons, reverse-engineered from a production Arbitrum/Stylus monorepo. End state: a skeleton that wires every layer (L0–L7) into jobs that each emit one manifest entry, terminating in the [release-gate.md](release-gate.md) step.

## 1. Top-level `permissions: {}` + per-job least-privilege grants

Every workflow declares **zero permissions at the top** and grants the minimum per job. The default `GITHUB_TOKEN` is otherwise read/write-everything; an empty top-level block plus narrow per-job grants means a compromised step can do nothing it wasn't explicitly allowed.

```yaml
permissions: {}          # top-level: deny all
jobs:
  lint:
    permissions:
      contents: read     # this job only reads the checkout
  kani:
    permissions:
      contents: write        # commits kani-status.json back
      pull-requests: write   # opens the ci-data PR
```

Real grants observed: `lint`/`test-*`/`frontend`/`subgraph`/`secrets-scan` → `contents: read`; `kani` and `loadtest` → `contents: write` + `pull-requests: write` (they commit CI-as-data); the keeper crons (`vigil-keeper`, `agents-cron`, `lantern-cron`) → top-level `permissions: {}` with `contents: read` on the working job. Never grant `write` to a job that only reads.

## 2. SHA-pinned actions

Every third-party action is pinned to a **full commit SHA**, with the version as a trailing comment. A moving tag (`@v4`) is a supply-chain hole — the tag can be repointed at malicious code. Pin the SHA; the comment documents intent.

```yaml
- uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332          # v4.1.7
- uses: pnpm/action-setup@fe02b34f77f8bc703788d5817da081398fad5dd2         # v4.0.0
- uses: actions/setup-node@1e60f620b9541d16bece96c5465dc8ee9832be0b        # v4.0.3
- uses: foundry-rs/foundry-toolchain@8f1998e9878d786675189ef566a2e4bf24869773 # v1.2.0
- uses: gitleaks/gitleaks-action@83373cf2f8c4db6e24b41c1a9b086bb9619e9cd3  # v2.3.7
- uses: peter-evans/create-pull-request@4320041ed380b20e97d388d56a7fb4f9b8c20e79 # v7.0.0
- uses: actions/upload-artifact@0b2256b8c012f0828dc542b3febcab082c67f72b   # v4.3.4
```

(The keeper crons that pin `@v4` / `@v1` — `pyth-keeper.yml` — are the exception, not the rule; harden them to SHAs.)

## 3. CI-as-data — status JSON regenerated in CI, committed back via a `ci-data/*` PR

Dashboards and status badges read JSON checked into the repo. That JSON must be **machine-generated by a CI job and committed back through a PR** — never hand-maintained. The pattern: a job runs the real check, writes a status file, and uses `peter-evans/create-pull-request` on a dedicated `ci-data/*` branch so the data update is reviewable and never mixed with code.

The Kani job builds `kani-status.json` from the actual proof outcomes (`steps.<id>.outcome`), then opens a PR:

```yaml
# ci.yml — kani job (abridged)
- name: Build kani-status.json
  if: always()
  run: |
    STATE="unknown"
    if [ "${{ steps.plinth.outcome }}" = "success" ] && [ "${{ steps.sigil.outcome }}" = "success" ]; then STATE="pass";
    elif [ "${{ steps.plinth.outcome }}" = "failure" ] || [ "${{ steps.sigil.outcome }}" = "failure" ]; then STATE="fail"; fi
    cat > apps/verify/public/kani-status.json <<EOF
    { "state": "$STATE", "passed": $PASSED, "total": $TOTAL,
      "last_run_at": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
      "proof_run_url": "https://github.com/${{ github.repository }}/actions/runs/${{ github.run_id }}" }
    EOF
- name: Create PR with kani-status update
  if: github.ref == 'refs/heads/main' && always()
  uses: peter-evans/create-pull-request@4320041ed380b20e97d388d56a7fb4f9b8c20e79 # v7.0.0
  with:
    branch: ci-data/kani
    title: "ci: kani-status.json data update"
    commit-message: "ci: refresh kani-status.json"
    delete-branch: true
    add-paths: |
      apps/verify/public/kani-status.json
```

Same shape in `loadtest-nightly.yml` (`build-report.mjs` → `apps/verify/public/loadtest/latest.json`, branch `ci-data/loadtest-nightly`) and `archive-weekly.yml` (`ci-data/${{ github.workflow }}`). The `last_run_at` + `proof_run_url` fields are what make the **evidence rule** in [release-gate.md](release-gate.md) checkable: a status that no run produced has no `proof_run_url`, so a green badge with no CI run behind it is RED.

## 4. Anti-honesty-theater — a CI-written file that a test fails if hand-edited

A status/claims file is only trustworthy if **it cannot be quietly hand-edited to look green**. The guard is a test that re-derives the truth from the filesystem/chain and fails if the committed file disagrees. `docs-claims.test.mjs` does this for documentation claims — it extracts every backticked file path and ASCII-tree directory from `README.md` and **fails CI if any referenced path does not exist on disk**:

```js
// scripts/docs-claims.test.mjs (abridged)
describe('README.md, every backticked file path exists', () => {
  it('all README file-path references resolve on disk', async () => {
    const text  = await readDoc('README.md');
    const paths = extractBacktickedPaths(text);              // `*.sol|rs|ts|...` tokens
    const missing = paths.filter((p) => !existsSync(resolve(REPO_ROOT, p)) && !FUTURE_STATE_PATHS.has(p));
    assert.deepEqual(missing, [], `README references paths that do not exist: ${missing.join(', ')}`);
  });
});
```

The same principle, applied to CI-as-data: a test re-runs the check and asserts the committed `kani-status.json` / `latest.json` matches — so editing the JSON to fake a pass turns the build red. Adjacent anti-erosion guards in the same spirit: the **Kani harness-count gate** (proof count can't silently drop below `docs/kani-baseline.txt`) and the **coverage floor gate** (`lcov.info` parsed against a ratcheting `FLOOR`). A status file you can hand-edit to green is decoration; a status file a test pins to ground truth is evidence.

## 5. Discord alerts on main failures

Every scheduled/long-running workflow ends with a guarded failure notifier. The guard matters: a bare `curl` to an unset webhook exits non-zero and **masks the real failure**, so check the secret is present first.

```yaml
- name: Notify ops on failure
  if: failure()
  run: |
    if [ -z "$DISCORD_OPS_WEBHOOK" ]; then
      echo "DISCORD_OPS_WEBHOOK not set; skipping failure notification."
      exit 0
    fi
    curl -X POST -H 'Content-Type: application/json' \
      -d "{\"content\":\"⚠️ ${{ github.workflow }} failed on ${{ github.ref_name }}: ${{ github.run_id }}\"}" \
      "$DISCORD_OPS_WEBHOOK"
  env:
    DISCORD_OPS_WEBHOOK: ${{ secrets.DISCORD_OPS_WEBHOOK }}
```

Formal-verification workflows scope the alert to main only (`if: failure() && github.ref == 'refs/heads/main'`) to avoid PR-branch noise, and some use `sarisia/actions-status-discord` for a richer card.

## 6. Concurrency groups + cancel-in-progress

Every workflow declares a concurrency group keyed by workflow + ref. The `cancel-in-progress` policy is **deliberate per workflow type**:

```yaml
# PR-test workflows: cancel superseded PR runs, but never cancel a push to main
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}
```

- **PR test runs** (`ci.yml`, `e2e.yml`): `cancel-in-progress: ${{ github.event_name == 'pull_request' }}` — a new push to a PR kills the stale run; a push to main is never cancelled.
- **Self-looping keepers** (`pyth-keeper`, `lantern-cron`): `cancel-in-progress: true` — a fresh dispatch must cancel the previous long-lived loop so runs never stack (see [uptime-keeper.md](uptime-keeper.md)).
- **Upptime / 5-min ticks / nightly loadtest**: `cancel-in-progress: false` — never cancel an in-flight probe or a partially-written data run.

## 7. Path-filtered triggers

Expensive jobs only run when relevant files change. Scope `paths:` to the layer's source so a docs-only PR doesn't spin up Playwright or Halmos.

```yaml
# e2e.yml
on:
  pull_request:
    paths:
      - 'apps/verify/**'
      - 'apps/verify/tests/**'
      - '.github/workflows/e2e.yml'
  schedule:
    - cron: '0 3 * * *'      # nightly full run regardless
  workflow_dispatch:

# halmos.yml — only when contracts or proofs change
on:
  pull_request:
    paths: [ "tests/halmos/**", "contracts/**" ]
```

Pair the PR-path filter with a scheduled (`cron`) full run so nothing is permanently skipped — the path filter trims PR cost, the cron guarantees coverage.

## Putting it together — L0–L7 → one manifest → the gate

Each layer job does its work and **emits one manifest fragment** (`{name,status,metric?,threshold?,direction?,required}` — schema in [release-gate.md](release-gate.md)) as an artifact; the final `gate` job downloads all fragments, assembles `qa-manifest.json`, and runs `qa-gate.mjs`. Its exit code is the merge gate.

```yaml
name: QA
on:
  push:        { branches: [main] }
  pull_request: { branches: [main] }

permissions: {}                                   # §1 deny-all top level
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }} # §6
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}

jobs:
  # ---- L0 static / lint / type ----
  lint:
    runs-on: ubuntu-latest
    permissions: { contents: read }               # §1 least privilege
    steps:
      - uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332 # v4.1.7  §2 SHA-pin
      - run: cargo clippy --workspace --all-targets -- -D warnings
      - run: pnpm -r lint && node scripts/check-banned-words.mjs
      - run: echo '{"name":"lint","status":"pass","required":true}' > entry.json
      - uses: actions/upload-artifact@0b2256b8c012f0828dc542b3febcab082c67f72b # v4.3.4
        with: { name: m-lint, path: entry.json }

  # ---- L1 unit + property (frontend + contracts) ----
  unit:
    runs-on: ubuntu-latest
    permissions: { contents: read }
    steps:
      - uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332 # v4.1.7
      - run: pnpm --filter @app/web test && cargo test --workspace --all-features
      - run: echo '{"name":"unit","status":"pass","required":true}' > entry.json
      - uses: actions/upload-artifact@0b2256b8c012f0828dc542b3febcab082c67f72b # v4.3.4
        with: { name: m-unit, path: entry.json }

  # ---- L1 formal (Kani) — also emits CI-as-data (§3) + anti-erosion gate (§4) ----
  formal:
    runs-on: ubuntu-latest
    permissions: { contents: write, pull-requests: write }
    steps:
      - uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332 # v4.1.7
      - run: |   # harness-count anti-erosion gate
          ACTUAL=$(grep -rE "^[[:space:]]*#\[kani::proof\]" programs/ --include="*.rs" | wc -l | tr -d ' ')
          BASELINE=$(cat docs/kani-baseline.txt | tr -d '[:space:]')
          [ "$ACTUAL" -lt "$BASELINE" ] && { echo "::error::kani proofs regressed $ACTUAL < $BASELINE"; exit 1; } || true
      - run: cargo kani
      # ... build status JSON + create-pull-request on ci-data/kani (see §3) ...
      - run: echo '{"name":"formal","status":"pass","required":true}' > entry.json
      - uses: actions/upload-artifact@0b2256b8c012f0828dc542b3febcab082c67f72b # v4.3.4
        with: { name: m-formal, path: entry.json }

  # ---- L2 integration + indexer · L3 e2e · L4 load · L5 lighthouse · L6 security ----
  # (one job each, same shape: do the work, write entry.json, upload as m-<layer>)
  # e2e uses §7 path filter; loadtest uses §3 ci-data + §6 cancel-in-progress:false.

  # ---- GATE — needs every layer, assembles the manifest, blocks on exit 1 ----
  gate:
    runs-on: ubuntu-latest
    needs: [lint, unit, formal]   # + integration, e2e, load, lighthouse, security
    permissions: { contents: read }
    steps:
      - uses: actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332 # v4.1.7
      - uses: actions/download-artifact@... # all m-* fragments
        with: { path: parts }
      - run: node tools/qa-gate/assemble.mjs parts/ > qa-manifest.json
      - name: Gate (exit 1 blocks the PR)
        run: node tools/qa-gate/qa-gate.mjs qa-manifest.json --report QA_PROOF.md
      - uses: actions/upload-artifact@0b2256b8c012f0828dc542b3febcab082c67f72b # v4.3.4
        if: always()
        with: { name: qa-proof, path: QA_PROOF.md }
```

L7 (uptime/keeper) is **not** in this graph — it runs on its own schedule, alerts via Discord, and contributes a **non-required** manifest entry that warns without blocking.
